# 에이전트와 함께 IR Kit 운영하기

코딩 에이전트(Claude Code, Codex, Cursor 등)는 이 UI의 **대등한 파트너**이지, 얹어 놓은 꼼수가 아닙니다. 에이전트는 `ir` 커맨드라인을 통해 이 인터페이스와 같은 규칙을 지키며 움직입니다 — 결정은 당신이, 실무는 에이전트가. 그것이 당신의 "10명의 IR 팀"입니다.

## 다운로드에서 실데이터 대시보드까지 — "ir start"
1. **키트 받기:** `git clone https://github.com/howieyoung/ir-kit && cd ir-kit`
2. **앱 실행:** `node server.js` → http://127.0.0.1:4820 — 샘플 회사로 즉시 작동.
3. **저장소 폴더에서 에이전트를 열고**(Claude Code, Codex, Cursor…) 말하세요: **"ir start"**. 요령은 이 한마디 — 에이전트가 모든 것(문서 **수신함 inbox** 포함)을 만들고, 프라이버시 원칙을 밝히고, 이후를 안내합니다.
4. **문서를 inbox에**(`ir-workspace/inbox/`) — SAFE, 은행 명세서, 캡테이블, 덱, 정리 안 된 그대로 괜찮습니다. 문서가 디스크 곳곳에 흩어져 있다면 지정 폴더를 `ir scan`(파일명만, 동의 없이는 아무것도 열지 않음).
5. **모든 것을 분류** — `ir sort`가 데이터룸의 올바른 카테고리로. 문서를 추가하면 **언제든 재실행**; 일회성이 아닌 상시 파일링 서비스입니다.
6. **동의하에 추출:** 실제 캡테이블, SAFE, 월별 재무 — 모든 숫자에 출처(파일+페이지)가 붙고, 모호한 것은 확인 목록으로. 절대 추측하지 않습니다.
7. **검증 후 당신이 확인:** `ir check`가 깨끗이 통과하면 출처와 대조하며 대시보드를 함께 확인 — 그 후에야 샘플 플래그가 꺼집니다. 문서를 주지 않으면 대시보드는 그냥 샘플 데이터를 유지합니다.

그다음부터는 월간 리듬 — 아래의 rituals와 스케줄로.

## IR 마스터로 가는 전체 여정
투자자 관계를 몰라도 됩니다 — 각 일을 평범한 말로 에이전트에게 위임하세요. 전체 흐름과 각 단계 뒤의 ritual:

| 원하는 것 | 에이전트에게 할 말 | Ritual |
|---|---|---|
| 실제 문서로 세팅 | **"ir start"** | `onboard.md` |
| 깔끔한 데이터룸 구축 | "체크리스트에 맞춰 내 데이터룸을 감사해 줘" | `data-room-audit.md` |
| **맞는 투자자 찾기 + 접촉** | "prompts/investor-sourcing.md를 읽고 우리에게 맞는 투자자를 찾아 줘" | `investor-sourcing.md` |
| 투자자 미팅 준비 | "[투자자]와의 미팅을 준비해 줘" | `meeting-prep.md` |
| 월간 업데이트 발송 | "이번 달 투자자 업데이트 초안을 써 줘" | `draft-update.md` |
| **이사회 운영** | "prompts/board-meeting.md를 읽고 [날짜] 이사회를 준비해 줘" | `board-meeting.md` |
| **주주총회 운영** | "prompts/shareholder-meeting.md를 읽고 주주총회를 준비해 줘" | `shareholder-meeting.md` |
| 라운드 시작 | "라운드를 모델링하고 파이프라인 계획을 세워 줘" | `round-kickoff.md` |

모든 것이 당신의 컴퓨터에서 실행되고, 실제 숫자를 인용하며, 당신 없이는 아무것도 발송하지 않습니다.

## 인터페이스: ir CLI
예제가 있는 전체 레퍼런스: [docs/CLI.md](docs/CLI.md).

```
# 시작
ir start              가이드 세팅 — 전부 구축, 단계 감지, 다음 행동 출력
ir sort               inbox 문서를 데이터룸 카테고리로 분류(언제든 재실행)

# 읽기
ir status [--json]    파생 지표 전부를 한 번에 — 먼저 현황 파악
ir check              테스트 스위트 — data/*.json을 직접 수정했다면 필수
ir model round --pre 12000000 --new 3000000    프라이스드 라운드 SAFE 전환

# 쓰기(정합성 강제)
ir close-month 2026-07 --saas 31000 --ads 14000 --payroll 34000 ...
ir safe add "Fund X" --principal 50000 --cap 8000000 --status Signed
ir prospect add "Acme Capital" --fit "leads pre-seed dev-tools" --source "<url>"
ir update draft | mark-sent

# 온보딩 & 산출물
ir scan <folders>     후보 재무 문서 — 파일명만, 절대 열지 않음
ir export board-pack | tearsheet | captable
ir schedule show      월간 rituals용 cron 설정
```

에이전트가 지키는 세 가지 규칙(당신도):
1. **JSON 직접 수정보다 `ir` 동사 우선** — `safe add` 한 번에 캡테이블 장부, CRM 커밋, 투자자 기록, 발송 리스트가 동기화됩니다. 수동 수정은 이들을 조용히 어긋나게 합니다.
2. **JSON을 직접 고쳤다면 반드시 `ir check`** — exit 1은 불변식 파괴; 먼저 고치세요.
3. **숫자를 지어내지 않는다.** 없는 데이터는 null로; 마감 안 된 달은 업데이트 초안이 막힙니다.

## Rituals (prompts/)
반복되는 IR 작업마다 [prompts/](prompts/)에 표준 프롬프트가 있습니다 — 복사, 괄호 채우기, 붙여넣기. 모두 CLI 동사를 경유합니다:

| 프롬프트 | 에이전트가 하는 일 |
|---|---|
| `onboard.md` | **여기서 시작** — 동의 게이트: 폴더 스캔, 데이터룸 정리, 숫자마다 출처를 달아 실데이터 입력 |
| `investor-sourcing.md` | 맞는 투자자 리서치, `ir prospect add`로 기록(적합도+출처), 맞춤 아웃리치 초안 — 절대 발송하지 않음 |
| `monthly-close.md` | `ir close-month` + 경고마다 설명, 지난 업데이트의 약속 점검 |
| `safe-signed.md` | `ir safe add` + 가드레일 리포트 + 당일 액션 |
| `draft-update.md` | `ir update draft`로 뼈대, 당신의 재료로 서사 작성 |
| `meeting-prep.md` | 한 장 브리프: 그 투자자와의 히스토리, 외워야 할 숫자, 상대의 가장 어려운 질문 3개 |
| `board-meeting.md` | `ir export board-pack` + 안건, 소집 통지, 사전 자료, 결의안, 회의록 템플릿 |
| `shareholder-meeting.md` | 주총 소집 통지, 안건, 결의안, 의결권 시트, 위임장, 회의록 |
| `data-room-audit.md` | 계층 체크리스트와 대조해 심각도순 시정 목록 출력 |
| `round-kickoff.md` | 라운드 모델링, CRM 파이프라인 시딩, 배치식 조달 계획 수립 |

## 스케줄에 올리기
기계적인 부분은 에이전트조차 필요 없습니다 — `ir update draft`는 결정적으로 동작합니다:

```
# crontab -e   (또는: ir schedule show)
0 9 1 * *  cd ~/ir-kit && ./bin/ir.js status          # 1일: 마감 리마인더
0 9 3 * *  cd ~/ir-kit && ./bin/ir.js update draft    # 3일: 초안이 리뷰 대기
0 9 * * 1  cd ~/ir-kit && ./bin/ir.js check           # 월요일: 데이터 정합성
```

그 위에 에이전트 실행을 얹어 서사(하이라이트/로우라이트)를 채우세요 — 방법과 안전 규칙은 [prompts/schedule-updates.md](prompts/schedule-updates.md). 초안은 절대 자동 발송되지 않습니다: 검토와 발송은 언제나 당신 몫입니다.

## 키트 확장하기
이 코드베이스는 의도적으로 에이전트가 고치기 쉽게 만들어졌습니다: 의존성 없는 순수 JS, 빌드 과정 없음. 에이전트에게 새 모듈("ESOP 추적 추가", "다중 통화 지원")을 요청하고 [AGENTS.md](AGENTS.md)를 가리키세요 — 규약은 문서화되어 있습니다: 새 연산은 `core/ops.js`에 넣고 CLI 동사를 추가, 수학은 `public/js/metrics.js`, 새 페이지는 `app.js`에 등록. 확장할 때도 약속을 지키세요: 의존성 제로, 로컬 우선, `ir check` 통과.

# 처음 30분

IR Kit의 루프는 하나뿐입니다: **일어난 일을 입력 → 대시보드가 핵심 숫자를 계산 → 업데이트가 투자자에게 전달.** 한 번 세팅하면 이후는 한 달에 한 시간 정도입니다.

> 코딩 에이전트를 쓰고 있나요? 아래 수동 입력은 대부분 건너뛸 수 있습니다 — **"ir start"**라고만 말하면, 매 단계 당신의 동의 아래 실제 문서로 시스템을 채웁니다. "에이전트와 사용" 탭을 보세요.

## 1. 내 것으로 만들기 (설정, 3분)
회사명, 창업자, 이메일, 라운드 목표, 업데이트 발송일을 설정하세요. 실제 데이터가 샘플을 완전히 대체하기 전까지 **샘플 플래그를 켜 두세요** — 사이드바의 노란 SAMPLE 배지가 "아직 진짜가 아니다"라고 알려 줍니다.

## 2. 재무 입력 (재무, 10분)
**기초 현금**(첫 달 이전의 은행 잔고)을 설정한 뒤, 한 달에 한 행씩: 매출 구성, 급여/인프라/기타 비용, 자금 유입, 인원수, 트랙션 지표. 자료가 없으면? 비워 두세요 — 절대 추측하지 않습니다. 계산 열(매출, 비용, 손익, 현금)과 대시보드는 입력과 동시에 갱신됩니다.

지난 6–12개월 자료가 있다면 그것부터: 히스토리가 MoM과 런웨이 숫자의 신뢰도를 만듭니다.

## 3. 캡테이블 입력 (캡테이블, 10분)
- **주주:** 창업자 주식, 부여된 옵션, 미배정 풀 — 기억이 아니라 설립 서류 기준으로.
- **SAFE 장부:** 체결한 모든 SAFE: 원금, 캡, 디스카운트, 날짜, 상태. 조건이 불확실하면 체결된 PDF를 확인하세요 — 자기 캡테이블에 대한 "잘 모르겠다"는 실사에서 가장 혹독하게 벌 받는 일입니다.
- 처음이라면 "캡테이블 101" 탭을 먼저.

## 4. 투자자 설정 (투자자 CRM, 5분)
기존 투자자를 추가하고, 업데이트를 받아야 할 모든 사람을 발송 리스트(세그먼트 포함)에 넣으세요. 라운드 진행 중이면 파이프라인을 커밋 탭에.

## 5. 첫 업데이트 발송 (투자자 업데이트, 상시)
작성기가 실제 지표를 자동으로 채웁니다 — 괄호를 채우고, 3분 안에 읽히게 줄이고, BCC 버튼으로 보낸 뒤 **발송 완료 → 보관**을 누르세요. 보관함이 핵심입니다: 미래의 리드 투자자는 히스토리를 처음부터 끝까지 읽고, 끊기지 않는 연속 기록은 스타트업이 가질 수 있는 가장 저렴한 신용 자산입니다.

## 이후의 월간 리듬
| 언제 | 무엇을 | 어디서 |
|---|---|---|
| 월말 + 최대 10일 | 월 마감 | 재무 |
| 같은 날 | 번/런웨이/경고 확인 | 대시보드 |
| 업데이트일(직접 설정) | 업데이트 발송 | 투자자 업데이트 |

"투자자 업데이트"에서 캘린더 파일을 내려받아 이 리듬을 실제 캘린더에 넣으세요.

# 캡테이블 101 — 10분 버전

캡테이블을 처음 보시나요? "캡테이블" 페이지를 자신 있게 쓰기 위한 최소한의 지식입니다. (계획용 지식이며 법률 자문이 아닙니다.)

## 기본
**캡테이블**은 "누가 얼마나 갖고 있는가"의 목록입니다. 보유량은 **주식 수**로 세고, 당신의 지분율 = 내 주식 수 ÷ 전체 주식 수 — 기준은 **완전 희석(fully diluted)**, 즉 아직 아무도 갖고 있지 않은 **옵션 풀**(미래 직원용 예약분)까지 분모에 넣습니다. 그것이 정직한 분모이고, 투자자가 쓰는 분모입니다.

## SAFE를 쉽게 말하면
**SAFE**는 "지금 돈, 나중에 주식"입니다: 투자자가 오늘 송금하고, 당신이 **프라이스드 라운드**를 하면 주식을 받습니다. 핵심 조건은 **밸류에이션 캡** — 표준 **post-money SAFE**에서 프라이스드 라운드 직전 투자자의 지분은 간단히:

**내재 지분 = 원금 ÷ post-money 캡** (예: $100K, 캡 $8M = 1.25%)

SAFE 장부가 "내재 %"를 보여 주는 이유입니다. 키트가 시각적으로 지키게 하는 두 가지 경험칙:
- 프라이스드 라운드 전에 **SAFE 내재 지분 합계를 약 15% 아래로** — post-money SAFE는 쌓이고, 그 희석은 **창업자 혼자 흡수**합니다.
- 모든 SAFE를 **같은 조건**으로. 캡이 제각각인 SAFE 더미는 실사의 악몽이며 확신 부족의 신호입니다.

**디스카운트**(예: 20%)는 SAFE를 신규 투자자보다 싼 가격에 전환시킵니다; 캡과 디스카운트가 둘 다 있으면 투자자에게 유리한 쪽이 적용됩니다.

## 라운드 모델러가 보여 주는 것
실제로 프라이스드 라운드를 하면 세 가지가 동시에 일어납니다: SAFE의 주식 전환, 옵션 풀의 목표 비율까지 **충전**, 신규 투자자의 진입. 모두의 지분율이 움직입니다. 모델러는 진짜 post-money SAFE 메커니즘으로 그 계산을 합니다 — 프리머니 밸류, 조달액, 풀 목표를 바꿔 가며 누가 무엇을 갖게 되는지 보세요. 창업자라면 **희석 시나리오** 탭을: 시리즈 A 진입 전에 약 50% 아래라면, 라운드 규모나 캡을 다시 생각할 때는 텀시트가 아니라 *지금*입니다.

## 엑싯 워터폴이 보여 주는 것
투자자는 보통 **1배 청산우선권**이 붙은 **우선주**를 보유합니다: 매각 시 투자금을 먼저 회수하*거나* 지분율대로 받거나 — 더 큰 쪽을 택합니다. 워터폴은 매각가별로 누가 얼마를 받는지 보여 줍니다. 보통주(당신과 팀)가 실제 돈을 만지기 시작하는 가격대까지 포함해서요. **MOIC**은 투자자의 투하자본 배수입니다.

## 어떤 투자자 미팅에서도 즉답해야 할 세 숫자
1. 오늘 기준 SAFE 내재 지분 합계.
2. 다음 프라이스드 라운드 후 창업자 지분율(기본 시나리오).
3. 런웨이(개월).

셋 다 이 키트에서 두 번의 클릭 안에 보입니다. 자기 캡테이블에 "확인해 보고 알려드릴게요" — 그 순간 미팅은 끝납니다.

# 용어집

| 용어 | 의미 |
|---|---|
| ARR(연환산 매출) | 당월 매출 × 12. 계약된 ARR과 다름 — 정직하게 표기 |
| Net burn(순소진) | 한 달의 비용 − 매출; 조달 유입 제외 |
| 캡(밸류에이션 캡) | SAFE 전환에 쓰는 밸류 — 캡이 낮을수록 투자자 주식이 많아짐 |
| 데이터룸 | 실사에서 투자자가 검토하는 정리된 문서 폴더 |
| 희석 | 투자자/풀에 신주가 발행되며 내 지분율이 줄어드는 것 — 내 *주식 수*는 그대로 |
| 디스카운트 | SAFE는 (1 − 디스카운트) × 라운드 가격으로 전환; 캡과 유리한 쪽 적용 |
| 완전 희석(FD) | 모든 옵션과 미발행 풀을 포함한 주식 수 — 정직한 분모 |
| 청산우선권 | 엑싯 시 우선주가 투자금을 먼저 회수(1x 비참가형 = 회수 또는 전환 중 하나만) |
| MFN | "최혜국" 사이드레터 — 나중 투자자가 더 좋은 조건을 받으면 자동 동일 적용. 비밀 특혜가 반드시 드러나는 이유 |
| MoM | 전월 대비 성장률. 평균을 말할 때는 평균 구간을 명시 |
| MOIC | 투하자본 배수 — 투자자가 받은 돈 ÷ 넣은 돈 |
| 옵션 풀 | 미래 팀을 위해 예약된 주식; 라운드마다 충전되며 보통 창업자를 희석 |
| Post-money SAFE | 표준(2018+) YC SAFE: 지분이 라운드 직전에 원금 ÷ 캡으로 고정 |
| Post-money / pre-money | 신규 자금 이후/이전의 기업 가치: post = pre + 조달액 |
| 프로라타 | 미래 라운드에서 지분율 유지를 위해 추가 투자할 수 있는 권리 |
| 프라이스드 라운드 | 합의된 주가로 하는 지분 투자(시드, 시리즈 A…) — SAFE가 여기서 전환 |
| Runway(런웨이) | 현금 ÷ 월평균 번 = 현금 소진까지 남은 개월 |
| SAFE | 미래 지분 취득 간이 계약 — 지금 돈, 다음 프라이스드 라운드에서 주식 |
| 사이드레터 | SAFE에 딸려 특정 투자자에게만 주는 추가 조건 — 실사에서 반드시 공개 |
| 409A | 보통주의 독립 평가; 미국에서 옵션 부여 전 필수 |
