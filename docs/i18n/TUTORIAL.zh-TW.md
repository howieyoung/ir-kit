# 與你的 Agent 一起操作 IR Kit

你的 coding agent（Claude Code、Codex、Cursor 等）是這個介面的**對等夥伴**，不是外掛補丁。它透過 `ir` 指令列運作，遵守與本介面相同的規則 — 你做決定，agent 做庶務。這就是你的「十人 IR 團隊」。

## 從下載到真實儀表板 — 說「ir start」
1. **取得工具組：** `git clone https://github.com/howieyoung/ir-kit && cd ir-kit`
2. **啟動 app：** `node server.js` → http://127.0.0.1:4820 — 內建範例公司，立即可用。
3. **在專案資料夾開啟你的 coding agent**（Claude Code、Codex、Cursor…）然後說：**「ir start」**。訣竅就這一句 — agent 會建好一切（含文件**收件匣 inbox**）、說明隱私底線，並一路引導你。
4. **把文件丟進 inbox**（`ir-workspace/inbox/`）— SAFE、銀行對帳單、股權表、簡報，完全不整理也沒關係。文件散落在硬碟各處？agent 可改用 `ir scan` 掃描你指定的資料夾（只看檔名，未經你同意不開啟任何檔案）。
5. **它會把所有文件歸檔**到正確的資料室分類，用 `ir sort` — 新增文件後**隨時可重跑**；這是常駐歸檔服務，不是一次性動作。
6. **經你同意後開始抽取：** 真實的股權結構表、SAFE、每月財務 — 每個數字都附文件出處；模糊之處進入待確認清單，絕不猜測。
7. **它驗證，你確認：** `ir check` 全數通過後，它帶你對照出處逐一檢視儀表板，之後範例旗標才會關閉。若你沒有提供文件，儀表板就維持範例資料。

之後就是每月節奏 — 見下方的 rituals 與排程。

## 完整 IR 大師之旅
你不需要懂投資人關係 — 用白話把每件事交辦給 agent。完整歷程與每一步背後的 ritual：

| 你想要… | 對 agent 說 | Ritual |
|---|---|---|
| 用真實文件完成建置 | **「ir start」** | `onboard.md` |
| 建立乾淨的資料室 | 「用檢查清單稽核我的資料室」 | `data-room-audit.md` |
| **找到合適的投資人並接觸** | 「讀 prompts/investor-sourcing.md，找適合我們的投資人」 | `investor-sourcing.md` |
| 準備投資人會議 | 「幫我準備與 [投資人] 的會議」 | `meeting-prep.md` |
| 寄出月度更新 | 「起草這個月的投資人更新」 | `draft-update.md` |
| **舉行董事會** | 「讀 prompts/board-meeting.md，籌備我 [日期] 的董事會」 | `board-meeting.md` |
| **舉行股東會／AGM** | 「讀 prompts/shareholder-meeting.md，準備我的股東會」 | `shareholder-meeting.md` |
| 啟動募資 | 「模擬我的輪次並建立 pipeline 計畫」 | `round-kickoff.md` |

一切都在你的機器上執行、引用你的真實數字，且未經你同意絕不寄出任何東西。

## 介面：ir CLI
完整參考與範例：[docs/CLI.md](docs/CLI.md)。

```
# 開始
ir start              引導式建置 — 建好一切、偵測進度、印出下一步
ir sort               將 inbox 文件歸類到資料室分類（隨時可重跑）

# 讀取
ir status [--json]    一次取得所有衍生指標 — 先掌握現況
ir check              測試套件 — 直接編輯 data/*.json 後必跑
ir model round --pre 12000000 --new 3000000    定價輪 SAFE 轉換

# 寫入（強制維持資料一致性）
ir close-month 2026-07 --saas 31000 --ads 14000 --payroll 34000 ...
ir safe add "Fund X" --principal 50000 --cap 8000000 --status Signed
ir prospect add "Acme Capital" --fit "leads pre-seed dev-tools" --source "<url>"
ir update draft | mark-sent

# onboarding 與產出
ir scan <folders>     找候選財務文件 — 只看檔名，絕不開啟
ir export board-pack | tearsheet | captable
ir schedule show      每月 rituals 的 cron 設定
```

Agent 遵守的三條規則（你也該遵守）：
1. **優先用 `ir` 指令而非直接改 JSON** — 一句 `safe add` 同步更新股權帳本、CRM 承諾、投資人紀錄與收件名單；手改很容易讓它們悄悄脫鉤。
2. **直接改過 JSON 後必跑 `ir check`** — exit 1 代表破壞了一致性；先修好再做別的。
3. **絕不捏造數字。** 缺的資料保持 null；未月結的月份無法產生更新信草稿。

## Rituals（prompts/）
每個週期性 IR 任務在 [prompts/](prompts/) 都有標準 prompt — 複製、填括號、貼上。它們都經由 CLI 指令執行：

| Prompt | Agent 做什麼 |
|---|---|
| `onboard.md` | **從這裡開始** — 逐步徵求同意：掃描資料夾、整理資料室、每個數字附出處地填入真實資料 |
| `investor-sourcing.md` | 研究合適投資人，逐一用 `ir prospect add` 記錄（適配度＋來源），起草個人化開發信 — 絕不寄出 |
| `monthly-close.md` | `ir close-month` ＋ 解釋每個警示、核對上個月更新信的承諾 |
| `safe-signed.md` | `ir safe add` ＋ 稀釋警戒線報告 ＋ 當日待辦 |
| `draft-update.md` | 用 `ir update draft` 產生骨架，再依你的素材撰寫敘事 |
| `meeting-prep.md` | 一頁簡報：與該投資人的歷史、必須背熟的數字、對方最難的三個問題 |
| `board-meeting.md` | `ir export board-pack` ＋ 議程、開會通知、會前資料、決議文與會議紀錄範本 |
| `shareholder-meeting.md` | 股東會／臨時股東會通知、議程、決議、股權表決表、委託書、會議紀錄 |
| `data-room-audit.md` | 對照分級檢查清單稽核資料室，依阻斷嚴重度輸出待辦清單 |
| `round-kickoff.md` | 模擬輪次、建立 CRM pipeline、產出批次式募資計畫 |

## 排上時程
機械性的部分甚至不需要 agent — `ir update draft` 是決定性的：

```
# crontab -e   （或：ir schedule show）
0 9 1 * *  cd ~/ir-kit && ./bin/ir.js status          # 每月 1 日：月結提醒
0 9 3 * *  cd ~/ir-kit && ./bin/ir.js update draft    # 每月 3 日：草稿等你審閱
0 9 * * 1  cd ~/ir-kit && ./bin/ir.js check           # 每週一：資料完整性
```

再疊一層 agent 排程來補上敘事（亮點／低谷）— 作法與安全規則見 [prompts/schedule-updates.md](prompts/schedule-updates.md)。草稿永不自動寄出：審閱與寄出永遠在你手上。

## 擴充工具組
這套程式碼刻意做成 agent 可直接編輯：零依賴的原生 JS、無建置流程。跟 agent 要新模組（「加 ESOP 追蹤」、「多幣別支援」）並指向 [AGENTS.md](AGENTS.md) — 慣例都寫好了：新操作放 `core/ops.js` 並加 CLI 指令、數學在 `public/js/metrics.js`、新頁面在 `app.js` 註冊。擴充時守住承諾：零依賴、本機優先、`ir check` 依然通過。

# 你的前 30 分鐘

IR Kit 只有一個循環：**輸入發生的事 → 儀表板算出關鍵數字 → 更新信告訴投資人。** 建置一次，之後每月大約一小時。

> 有 coding agent？下面的手動輸入大多可以跳過 — 只要跟它說 **「ir start」**，它就會在你每一步同意下，用你的真實文件填好整個系統。詳見「搭配 Agent 使用」分頁。

## 1. 變成你的（設定，3 分鐘）
填入公司名稱、創辦人、Email、募資目標、更新信寄出日。在真實資料完全取代範例之前，**保持範例旗標開啟** — 側欄的黃色 SAMPLE 標籤提醒你：這裡的數字還不是真的。

## 2. 輸入財務（財務，10 分鐘）
先設定**期初現金**（第一個月之前的銀行餘額），然後每月一列：營收拆分、薪資／基礎設施／其他成本、融資入金、人數、以及你的 traction 數據。缺資料？留空 — 絕不用猜的。計算欄位（營收、成本、損益、現金）和儀表板會隨輸入即時更新。

有過去 6–12 個月的資料就先填：歷史紀錄是 MoM 與跑道數字可信度的來源。

## 3. 輸入股權結構（股權結構表，10 分鐘）
- **股東：** 創辦人股數、已配發選擇權、未配發選擇權池 — 依據公司設立文件，不是憑記憶。
- **SAFE 帳本：** 每一份簽過的 SAFE：本金、估值上限、折扣、日期、狀態。條款不確定就翻出已簽署的 PDF 確認 — 對自己的股權表「不確定」是盡職調查最嚴懲的事。
- 第一次接觸？先讀「股權結構入門」分頁。

## 4. 建立投資人（投資人 CRM，5 分鐘）
先加入現有投資人，再加入所有應收到更新信的人（更新信收件名單分頁，含分眾）。募資中的話，把 pipeline 填進募資承諾。

## 5. 寄出第一封更新信（投資人更新，長期）
撰寫器會預填你的真實指標 — 填好括號、刪到三分鐘可讀完，用 BCC 按鈕寄出，然後按**標記已寄出 → 歸檔**。歸檔才是重點：未來的領投人會把整份歷史一次讀完，不中斷的連續紀錄是新創最便宜的信用資產。

## 之後的每月節奏
| 何時 | 做什麼 | 在哪裡 |
|---|---|---|
| 月底後最多 10 天 | 月結 | 財務 |
| 同一天 | 檢查燒錢／跑道／警示 | 儀表板 |
| 更新日（自訂） | 寄出更新信 | 投資人更新 |
| 任何 SAFE 簽署 | 帳本 + CRM + 收件名單，當天完成 | 股權結構表 + CRM |
| 每季 | 董事會／投資人會議資料包 | 攻略手冊 → 董事會資料包 |

到「投資人更新」下載行事曆檔，把這個節奏放進你真正的行事曆。

# 股權結構入門 — 十分鐘版

沒看過 cap table？這是自信使用「股權結構表」頁面的最低知識量。（規劃用知識，不是法律意見。）

## 基礎
**股權結構表（cap table）**就是「誰擁有多少」的清單。持有以**股數**計；你的百分比＝你的股數 ÷ 全部股數 — 以**完全稀釋（fully diluted）**為基準，也就是把**選擇權池**（保留給未來員工的股份）也算進分母，即使還沒有人持有。這才是誠實的分母，也是投資人用的那個。

## 用白話說 SAFE
**SAFE** 是「現在給錢、以後給股」：投資人今天匯款，等你進行**定價輪**時換成股份。關鍵條款是**估值上限（cap）** — 在標準的 **post-money SAFE** 下，投資人在定價輪前一刻的持股就是：

**隱含持股 ＝ 本金 ÷ post-money 估值上限**（例：$100K、上限 $8M ＝ 1.25%）

這就是 SAFE 帳本顯示「隱含 %」的原因。工具組會用視覺提醒兩條經驗法則：
- 定價輪之前，**SAFE 隱含持股總和壓在約 15% 以下** — post-money SAFE 會疊加，而且**稀釋全部由創辦人吸收**。
- 每份 SAFE **條款一致**。一疊不同 cap 的 SAFE 是盡職調查的災難，也顯得說服力不足。

**折扣**（如 20%）讓 SAFE 用比新投資人便宜的價格轉換；cap 與折扣並存時，投資人取對自己較有利者。

## 定價輪模擬器在算什麼
當你真的進行定價輪，三件事同時發生：SAFE 轉成股份、選擇權池通常**補足**到目標比例、新投資人進場。所有人的百分比都會變動。模擬器用真實的 post-money SAFE 機制計算 — 調整 pre-money 估值、募資額、池目標，看每個人最後拿到什麼。創辦人請看**稀釋情境**分頁 — 若進入 A 輪前就低於約 50%，該重想輪次規模或 cap 的時候是*現在*，不是拿到 term sheet 之後。

## 出場瀑布在算什麼
投資人通常持有**特別股**，帶 **1 倍清算優先權**：出售時，他們先拿回本金*或*按持股比例分 — 取金額較高者。瀑布圖顯示不同售價下每個人分多少，包括普通股（你和團隊）從哪個價位開始真正拿到錢。**MOIC** 是投資人的投入資本回報倍數。

## 任何投資人會議都要背熟的三個數字
1. 今天的 SAFE 隱含持股總和。
2. 下一個定價輪後的創辦人持股比例（基準情境）。
3. 你的跑道（月數）。

三個數字在這套工具裡兩次點擊內都看得到。對自己的股權表說「我回去查一下」，會議就結束了。

# 名詞解釋

| 名詞 | 意義 |
|---|---|
| ARR run-rate（年化營收） | 當月營收 × 12。與已簽約 ARR 不同 — 標示要誠實 |
| Net burn（淨燒錢） | 當月成本減營收；不含融資入金 |
| Cap（估值上限） | SAFE 轉換用的估值 — 上限越低，投資人拿到的股份越多 |
| 資料室（Data room） | 供投資人盡職調查審閱的文件資料夾 |
| 稀釋 | 新股發給投資人／選擇權池時你的百分比縮小 — 你的*股數*不變 |
| 折扣 | SAFE 以（1 − 折扣）× 輪次價格轉換；投資人取 cap 或折扣中較優者 |
| 完全稀釋（FD） | 含所有選擇權與未發池的股數 — 誠實的分母 |
| 清算優先權 | 出場時特別股先拿回本金（1x 不參與＝拿回本金*或*轉換，擇一） |
| MFN | 「最惠國」條款 side letter — 後來投資人拿到更好條件時，此投資人自動比照。秘密特殊條件終會曝光的原因 |
| MoM | 月增率。引用平均值時要說明平均區間 |
| MOIC | 投入資本回報倍數 — 投資人拿回的 ÷ 投入的 |
| 選擇權池 | 保留給未來團隊的股份；各輪補足，通常稀釋創辦人 |
| Post-money SAFE | 標準（2018 後）YC SAFE：持股在定價輪前一刻鎖定為本金 ÷ cap |
| Post-money / pre-money | 新資金進來後／前的公司估值：post ＝ pre ＋ 募資額 |
| Pro-rata | 投資人在未來輪次追加投資以維持持股比例的權利 |
| 定價輪 | 以議定股價進行的股權融資（種子輪、A 輪…）— SAFE 在此轉換 |
| Runway（資金跑道） | 現金 ÷ 月均燒錢 ＝ 距離現金歸零的月數 |
| SAFE | 未來股權簡單協議 — 現在拿錢，下個定價輪給股 |
| Side letter | 隨 SAFE 給予單一投資人的額外條款 — 盡職調查中必須揭露 |
| 409A | 普通股的獨立估值；美國發選擇權前必備 |
